# Kit conventions (AUTHORITATIVE ‚Äî read this first)

The other references are forked from skybridge's generic skill and are mostly right, but **where they differ from
the kit's actual files, the KIT WINS.** This page captures the kit-specific truth + the things the generic docs
miss. When in doubt, copy the patterns in the example `src/server.ts` / `src/views/tasks.tsx` / `src/store.ts`.

## 1. Server entry (don't follow the generic snippet)
The generic docs show `server.run();` then a type export. The kit's worker entry needs the **default export**:
```ts
const server = new McpServer({ name: "your-app", version: "0.1.0" }, { capabilities: {} })
  .registerTool(/* ‚Ä¶ */);
export default await server.run();   // ‚Üê the worker's fetch handler. NOT `server.run();` (void).
export type AppType = typeof server; // ‚Üê powers typed useToolInfo/useCallTool in helpers.ts
```

## 2. registerTool ‚Äî always set `outputSchema`
Every tool sets `outputSchema` (the generic fetch-and-render snippet omits it). It validates `structuredContent`
**and** types `useToolInfo().output` in the view. A view-bearing tool also sets `view`:
```ts
.registerTool(
  { name: "show_x", description: "Render ‚Ä¶", inputSchema: {}, annotations: { readOnlyHint: true },
    outputSchema: { items: z.array(ItemSchema) },
    view: { component: "x" } },          // ‚Üí src/views/x.tsx (file name === component)
  async () => ({ structuredContent: { items: await (await getStore()).list() }, content: [{ type: "text", text: "‚Ä¶" }] }),
)
```
**One view backs exactly one tool** (skybridge enforces it). Render with one view-bearing tool; do mutations with
**non-view** tools (`add_*`/`toggle_*`/`remove_*`) the view calls back via `useCallTool`.

### The view-name registry ‚Äî run dev/build after adding/renaming a view
`view.component` is type-checked against a **build-generated** `.skybridge/views.d.ts` (the `ViewNameRegistry`).
It's regenerated by the view-scan when you run `npm run dev` or `skybridge build`. So after you add or rename a
view file, **run dev/build before you typecheck** ‚Äî otherwise `registerTool({ view: { component: "habits" }})`
fails with a misleading `Type '"habits"' is not assignable to type '"tasks"'`. That error is stale codegen, not a
real bug ‚Äî start the dev server and it clears.

### Mutation ‚Üí re-render (the core data flow ‚Äî learn it here, not by guessing)
The idiomatic loop for "user did something in the view": the view calls a mutation tool with **`callToolAsync`**,
then re-seeds its local state from the returned `structuredContent`:
```tsx
const add = useCallTool("add_x");
const onAdd = async (title: string) => {
  const res = await add.callToolAsync({ title });
  setItems(res.structuredContent.items);   // re-seed from the tool's fresh result
};
```
(Claude can also just call the mutation tool itself in chat, then call `show_x` to re-render ‚Äî same effect.)

## 3. The view contract + `viewUUID`
A view-bearing tool's result carries a **fresh `_meta.viewUUID` on every call** ‚Äî that's how the host re-mounts
the view. So calling `show_x` again after a mutation re-renders with the new `structuredContent`. (`viewUUID` is
also your success signal when verifying on the wire ‚Äî see ¬ß6.) `structuredContent` is what the **LLM** sees; keep
view-only/bulky data out of it (`_meta`).

## 4. View hooks (from `../helpers.js` + `skybridge/web`)
- `useToolInfo<"show_x">()` ‚Üí `.output` (your `outputSchema` shape), `.isPending`.
- `useCallTool("add_x")` ‚Üí `callTool(args)` (fire-and-forget; check `.isPending`/`.data`) **or**
  `await callToolAsync(args)` ‚Üí returns the result so you can re-seed local state from `result.structuredContent`.
  The example uses `callToolAsync` for the "mutate ‚Üí re-seed" pattern ‚Äî prefer it.
- `useSendFollowUpMessage()` ‚Üí `send("‚Ä¶")` to steer the assistant from a view action (see prompt-llm.md).

## 5. Renaming the DO (4 strings must agree ‚Äî easy to get a silent fallback wrong)
When you rename `TaskDO` ‚Üí `YourDO`, change **all four** or the store silently falls back to in-memory (invisible
under `skybridge dev` ‚Äî you'll only notice under `dev:do`):
1. `src/store.ts` ‚Äî the probe `mod.env?.TASK_DO` ‚Üí `mod.env?.YOUR_DO`
2. `wrangler.jsonc` ‚Äî the binding `"name": "TASK_DO"` ‚Üí `"YOUR_DO"`
3. `wrangler.jsonc` ‚Äî `"class_name": "TaskDO"` + `migrations[].new_sqlite_classes: ["TaskDO"]` ‚Üí `YourDO`
4. `cf-entry.ts` ‚Äî `export { TaskDO } from "./src/task-do.js"` ‚Üí `YourDO`

(`grep -rn TaskDO src cf-entry.ts wrangler.jsonc` after renaming to catch stragglers.)

## 6. Verify on the wire (works without Claude Desktop)
The MCP endpoint **requires** the `Accept: application/json, text/event-stream` header (else 406). Sanity-check a
tool without an LLM:
```bash
curl -s -X POST http://localhost:3010/mcp \
  -H 'Content-Type: application/json' -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"show_x","arguments":{}}}'
```
A view tool's result has `structuredContent` + a fresh `_meta.viewUUID` (call twice ‚Üí different UUIDs = working).
Under `wrangler dev` the body may be SSE ‚Äî grab the last `data:` line. Also: `tools/list` shows your tools +
the `ui.resourceUri`/`openai/outputTemplate` meta on view-bearing ones.

## 7. Don't chase telemetry noise
The dev server logs `PostHog ‚Ä¶ ECONNREFUSED / fetch failed` (skybridge telemetry, fails offline). **Benign ‚Äî ignore
it.** It is NOT an app error.

## 8. Deployability: views use INLINE STYLES, not tailwind
To deploy to PMC, the view is bundled to a bare **ESM module** exporting `mount(el, ctx)` via esbuild (`deploy.mjs`;
React externalized to the PMC shell's instance, skybridge hooks fed from `ctx` by a deploy-only shim ‚Äî the
component + `skybridge dev` loop are unchanged) ‚Äî there's no vite/tailwind pipeline there. So **kit views must
style with an inline style object, not tailwind classes or CSS imports** (see `src/views/tasks.tsx`'s `styles()`
object). Inline styles render identically under `skybridge dev`, so this costs nothing locally and keeps the view
deployable (~5KB). **To deploy, FIRST ask the user for their PMC URL** ‚Äî
it is never hardcoded (usually a remote `https://‚Ä¶`; `localhost` only for local dev) ‚Äî then
`node deploy.mjs --base <their-PMC-url> --slug <unique-slug>` (splits the facet ‚Üí `modules`, the view ‚Üí `assets`/R2;
pushes via `app.builder.deploy`; `tasks` is a reserved slug). See `DEPLOY.md` ‚Äî note the facet
(`deploy-app/facet.js`, PMC `App extends DurableObject` format) currently mirrors the local app by hand.

## 9. Discovery mode
discover.md says to validate the SPEC with the user phase-by-phase. If the app is already decided (the user gave a
clear ask, or this is a non-interactive build), write the SPEC.md from the ask and proceed ‚Äî don't stall waiting
for validation that won't come.
